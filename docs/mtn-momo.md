# MTN MoMo Integration — Keshless Tickets API

## Overview

MTN Mobile Money (MoMo) is a payment method gated behind the `MTN_MOMO_ENABLED` env var and also toggleable via the admin dashboard payment-method settings. **Both** the env flag AND the dashboard toggle must be on for MoMo to be active.

`MtnMomoClient.isConfigured()` returns `false` when credentials are absent, so the rest of the API runs safely while MoMo is disabled.

---

## Env Vars

### Non-secret vars (already in `cloudbuild.yaml` `--set-env-vars`)

| Var | Current value (sandbox) | Notes |
|-----|------------------------|-------|
| `MTN_MOMO_ENABLED` | `false` | Flip to `true` at go-live |
| `MTN_MOMO_BASE_URL` | `https://sandbox.momodeveloper.mtn.com` | Change for prod — see below |
| `MTN_MOMO_TARGET_ENV` | `sandbox` | Change for prod — see below |
| `MTN_MOMO_CURRENCY` | `EUR` | Sandbox requires EUR; prod uses `SZL` |
| `MTN_MOMO_CALLBACK_URL` | _not yet set_ | Set at go-live: `https://<api-domain>/api/momo/callback` |

### Secret vars (NOT yet in cloudbuild — add at go-live, see below)

| Var | Secret Manager name | Description |
|-----|--------------------|----|
| `MTN_MOMO_SUBSCRIPTION_KEY` | `keshless-tickets-mtn-momo-sub-key` | Obtained from MTN MoMo Developer Portal |
| `MTN_MOMO_API_USER` | `keshless-tickets-mtn-momo-user-id` | UUID generated by sandbox provisioning script |
| `MTN_MOMO_API_KEY` | `keshless-tickets-mtn-momo-api-key` | Generated by sandbox provisioning script |

## One-Time Sandbox Provisioning

Before going live you must provision an API user and API key against the MTN sandbox. The script `scripts/provision-momo-sandbox.ts` does this automatically. Run it with the subscription key you got from the MTN MoMo Developer Portal:

```bash
MTN_MOMO_SUBSCRIPTION_KEY=<your-subscription-key> \
MTN_MOMO_BASE_URL=https://sandbox.momodeveloper.mtn.com \
npx ts-node scripts/provision-momo-sandbox.ts
```

The script prints `MTN_MOMO_API_USER` (a UUID) and `MTN_MOMO_API_KEY`. Save these — they go into Secret Manager at go-live.

## Go-Live Steps (when credentials are ready)

### 1. Create secrets in Secret Manager (deployer SA)

```bash
gcloud config configurations activate deployer
gcloud config set project contracts-470406

# Subscription key from MTN MoMo Developer Portal
echo -n "<SUBSCRIPTION_KEY>" | gcloud secrets create keshless-tickets-mtn-momo-sub-key \
  --data-file=- --project=contracts-470406

# API user UUID from provisioning script
echo -n "<API_USER_UUID>" | gcloud secrets create keshless-tickets-mtn-momo-user-id \
  --data-file=- --project=contracts-470406

# API key from provisioning script
echo -n "<API_KEY>" | gcloud secrets create keshless-tickets-mtn-momo-api-key \
  --data-file=- --project=contracts-470406
```

### 2. Grant secretAccessor to the Cloud Run runtime SA

First, look up the ACTUAL runtime SA for the `keshless-tickets-api` Cloud Run service:

```bash
gcloud run services describe keshless-tickets-api \
  --region=us-central1 \
  --project=contracts-470406 \
  --format='value(spec.template.spec.serviceAccountName)'
```

Then grant access for each secret:

```bash
RUNTIME_SA="<output-from-above>"

for SECRET in keshless-tickets-mtn-momo-sub-key keshless-tickets-mtn-momo-user-id keshless-tickets-mtn-momo-api-key; do
  gcloud secrets add-iam-policy-binding $SECRET \
    --project=contracts-470406 \
    --member="serviceAccount:${RUNTIME_SA}" \
    --role="roles/secretmanager.secretAccessor"
done
```

### 3. APPEND to `cloudbuild.yaml` `--set-secrets`

In `cloudbuild.yaml`, find the existing `--set-secrets` line:

```
'MONGODB_URI=keshless-tickets-mongodb-uri:latest,JWT_SECRET=keshless-tickets-jwt-secret:latest,SERVICE_KEY=keshless-tickets-service-key:latest'
```

Append (do NOT replace — use `--update-secrets` or append to the comma-separated string):

```
MTN_MOMO_SUBSCRIPTION_KEY=keshless-tickets-mtn-momo-sub-key:latest,MTN_MOMO_API_USER=keshless-tickets-mtn-momo-user-id:latest,MTN_MOMO_API_KEY=keshless-tickets-mtn-momo-api-key:latest
```

Final `--set-secrets` value:

```
MONGODB_URI=keshless-tickets-mongodb-uri:latest,JWT_SECRET=keshless-tickets-jwt-secret:latest,SERVICE_KEY=keshless-tickets-service-key:latest,MTN_MOMO_SUBSCRIPTION_KEY=keshless-tickets-mtn-momo-sub-key:latest,MTN_MOMO_API_USER=keshless-tickets-mtn-momo-user-id:latest,MTN_MOMO_API_KEY=keshless-tickets-mtn-momo-api-key:latest
```

### 4. Update non-secret env vars in `--set-env-vars`

Flip the flag and set the callback URL:

- `MTN_MOMO_ENABLED=true`
- `MTN_MOMO_CALLBACK_URL=https://<api-domain>/api/momo/callback`

### 5. Enable the admin dashboard toggle

The admin dashboard payment-method toggle also gates MoMo. Both `MTN_MOMO_ENABLED=true` AND the dashboard toggle must be on for MoMo to be live.

## Sandbox → Production Cutover

When cutting over to production MTN MoMo (real money):

| Var | Sandbox value | Production value |
|-----|--------------|-----------------|
| `MTN_MOMO_BASE_URL` | `https://sandbox.momodeveloper.mtn.com` | `https://proxy.momoapi.mtn.com` |
| `MTN_MOMO_TARGET_ENV` | `sandbox` | `mtneswatini` |
| `MTN_MOMO_CURRENCY` | `EUR` | `SZL` |
| `MTN_MOMO_SUBSCRIPTION_KEY` | Sandbox key from portal | Production key from MTN portal |

The `MTN_MOMO_API_USER` and `MTN_MOMO_API_KEY` must also be reprovisioned against the production environment using the production subscription key. Re-run `scripts/provision-momo-sandbox.ts` pointing at the production base URL, or provision manually via the MTN MoMo API.

Update secrets in Secret Manager with the new production values and redeploy.
